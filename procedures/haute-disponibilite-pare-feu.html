1 Contexte & objectif

Le protocole CARP (Common Address Redundancy Protocol) permet à plusieurs équipements réseau de partager une même adresse IP. C'est une alternative libre au protocole HSRP. Les machines partageant cette IP forment un groupe de redondance, avec un équipement « maître » et un ou plusieurs « esclaves » destinés à prendre le relai en cas de défaillance du premier.

Ici, CARP est utilisé pour créer un routeur / pare-feu redondant : les clients prennent l'IP virtuelle comme passerelle par défaut. En cas de panne du routeur principal, le second répond aux requêtes des clients en toute transparence, sans coupure. Le protocole pfsync gère quant à lui la synchronisation de configuration entre les deux pare-feu.

2 Prérequis & infrastructure

  • 2 pare-feu pfSense avec 4 réseaux différents chacun (WAN, CARP/pfsync, LAN, DMZ).
  • Un client Windows 10 dans le réseau LAN pour la configuration et les tests.
  • Un serveur web Debian dans le réseau DMZ pour les tests depuis l'extérieur.
  • Environnement entièrement virtualisé sous Hyper-V.
  • Une configuration de base pfSense déjà en place sur les deux pare-feu (voir la procédure « Pare-feu pfSense »).

5 Points d'attention & retour d'expérience

→
Toujours effectuer les modifications de configuration sur le pare-feu maître : elles sont répliquées automatiquement vers l'esclave via pfsync.
→
La valeur skew la plus basse désigne le maître : bien la vérifier sur chaque interface virtuelle (LAN, WAN, DMZ).
→
Sous Hyper-V, l'usurpation d'adresse MAC doit être activée sur les cartes réseau des VM pour que les IP virtuelles CARP répondent correctement.
→
Le NAT Outbound doit repasser en mode manuel dès qu'une interface virtuelle WAN est utilisée, pour translater uniquement le trafic voulu.
